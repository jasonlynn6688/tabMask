# 标签页伪装插件需求与技术方案

## 1. 背景

浏览网页时，Chrome 标签页会直接显示当前页面标题。该标题通常会暴露正在浏览的内容，容易被旁人通过浏览器标签栏直接识别。

本项目希望实现一个 Chrome 插件，用于对当前普通网页标签页标题进行自定义或伪装，并同步替换 favicon，降低旁观者从标签栏直接识别浏览内容的可能性。

## 2. 目标

- 允许用户对当前普通网页标签页名称进行自定义。
- 在页面自身更新标题时，仍尽量维持用户设定的标题。
- 支持绝大多数普通 `http/https` 网页。
- 提供最小可用版本，先解决“标签标题暴露当前浏览内容”的问题。

## 3. 非目标

以下内容不属于首个版本必须解决的范围：

- 隐藏页面主体中的视频内容、推荐列表、评论区等页面信息。
- 绕过网站安全策略或规避平台检测。
- 完全模拟另一个网站的全部页面行为。
- 支持 Chrome 内置页、Chrome Web Store、扩展页等受限页面。

## 4. 用户场景

### 4.1 核心场景

用户在办公室、图书馆或共享屏幕环境中观看 YouTube 视频，不希望标签页直接暴露视频标题。

### 4.2 使用方式

用户打开任意普通网页后：

- 点击插件图标。
- 输入一个自定义标题，或选择一个预设标题。
- 当前标签页名称被改成用户指定内容。

### 4.3 页面切换场景

用户在当前 tab 内继续浏览或页面脚本重写标题时，插件应继续维持伪装标题，而不是恢复为真实页面标题。

## 5. 需求说明

## 5.1 功能需求

### FR-1 自定义当前标签标题

插件应允许用户对当前普通网页页面设置一个自定义标签标题。

### FR-2 标题持续生效

当 YouTube 页面因前端脚本更新 `document.title` 时，插件应能重新覆盖标题，尽量保持用户设置值不被替换。

### FR-3 支持页面内跳转

页面在不整页刷新的情况下变更标题或路由时，插件应识别页面状态变化，并继续应用伪装标题。

### FR-4 作用范围受控

插件应只对普通网页生效，避免在受限页面上出现误导性操作。

### FR-5 可关闭伪装

用户应能恢复为网站默认标题，避免插件长期影响页面显示。

## 5.2 可选增强需求

### ER-1 预设标题

插件可提供若干一键伪装标题，例如“工作台”“文档中心”“项目管理”等。

### ER-2 favicon 伪装

插件可尝试替换当前页面 favicon，降低只看图标即可识别当前站点的风险。

### ER-3 记忆规则

插件可记录用户上次输入的标题，或为特定域名自动套用某个伪装标题。

## 5.3 非功能需求

- 插件交互应简单，最好在 1 到 2 步内完成伪装。
- 对页面性能影响要低，避免高频无意义重写标题。
- 不应在所有站点全局注入复杂逻辑。
- 插件权限尽量最小化。

## 6. 技术可行性结论

结论：可行。

Chrome 扩展本身没有直接通过 `chrome.tabs.update()` 修改标签标题的能力，但可以通过内容脚本运行在页面上下文中，修改页面的 `document.title`，从而间接影响浏览器标签页显示名称。

因此，该插件的核心实现不应依赖“直接修改浏览器 tab 元数据”，而应采用“在目标页面内接管标题显示”的方式实现。

## 7. 技术方案

## 7.1 方案总览

首版建议采用 Manifest V3 架构，包含以下模块：

- `manifest.json`
- `service worker`
- `content script`
- `popup` 配置面板
- 本地存储

推荐架构职责如下：

- `popup`
  - 接收用户输入的伪装标题。
  - 展示预设标题与开关操作。
- `service worker`
  - 协调 popup 与目标 tab 的消息通信。
  - 管理规则存储与页面注入。
- `content script`
  - 运行于被用户点击授权的当前网页。
  - 修改 `document.title`。
  - 监听页面标题变化、URL 变化，并重新应用伪装值。
- `chrome.storage.local`
  - 存储当前配置，例如是否启用、用户输入标题、预设方案等。

## 7.2 核心实现思路

### 步骤 1：用户输入伪装标题

用户点击插件图标，在 popup 中输入标题，例如“项目文档”。

### 步骤 2：把标题规则发送给当前标签页

popup 将标题值发送给后台脚本或直接发送给当前 tab 的 content script。

### 步骤 3：页面内覆盖 `document.title`

content script 在页面执行：

- 读取用户设置的目标标题。
- 将 `document.title` 改写为目标值。

### 步骤 4：监听并重复覆盖

由于网页自身可能会主动重写标题，content script 需要持续监听并在必要时重新设置标题。

可采用以下机制组合：

- `MutationObserver`
  - 监听 `document.querySelector("title")` 或文档头部变化。
- 定时校验
  - 低频校验 `document.title` 是否被还原。
- 历史状态监听配合后台事件
  - 处理站点的单页路由切换。

## 7.3 处理 YouTube 单页应用行为

部分站点常通过前端路由切换内容，不一定触发整页刷新。对应策略如下：

- 监听 URL 变化。
- 监听 `popstate`、`pushState`、`replaceState` 相关行为。
- 必要时结合扩展侧的导航事件感知页面已切换到新的 `watch` 页面。

当用户在当前 tab 中切换到新内容时，应触发：

1. 重新判断当前页面是否仍在支持范围。
2. 若已启用伪装，则继续把标题改写为用户设定值。

## 7.4 favicon 伪装方案

当前实现采用内容脚本动态替换页面 favicon：

- 查找页面中已有的 `<link rel="icon">` 或相关图标节点。
- 动态生成一个中性的 SVG “文档页”图标，不使用任何现成品牌 logo。
- 将当前页面 favicon 替换为该中性图标。
- 在网站重写 favicon 时，重新应用伪装图标。
- 在用户点击恢复时，还原最近一次观察到的原始 favicon 集合。

注意事项：

- 某些网站会重建 favicon 节点，需要重复覆盖。
- favicon 更换不隐藏页面主体内容，只降低标签栏识别度。

## 7.5 权限建议

首版尽量最小权限：

- `storage`
- `tabs`
- `activeTab`
- `scripting`

当前实现采用 `activeTab` 与 `chrome.scripting.executeScript()` 动态注入内容脚本，只在用户点击扩展后对当前活动 tab 获得临时访问权。

## 8. MVP 建议

建议先实现最小版本：

### MVP 范围

- 支持普通 `http/https` 网页
- 排除 Chrome 内置页、Chrome Web Store、扩展页等受限页面
- popup 中允许输入自定义标题
- 点击应用后立即修改当前页面标题
- 点击应用后同时替换当前页面 favicon 为中性文档图标
- 页面切换视频后继续保持伪装标题
- 页面重写 favicon 后继续保持中性图标
- 提供“恢复默认标题”按钮
- 提供恢复原 favicon 的能力

### MVP 暂不包含

- 多站点支持
- 多规则管理
- 复杂自动化策略

## 9. 风险与限制

### 9.1 只能隐藏标签标题，不能隐藏页面内容

即使 tab 标题被改掉，页面本身依然显示网页主体内容。

### 9.2 favicon 可能暴露站点身份

如果不改 favicon，旁人仍可能通过图标认出当前站点。

### 9.3 音频播放状态可能暴露

浏览器可能在标签页上显示播放状态、音量图标或媒体控制提示，这部分不一定能通过标题伪装解决。

### 9.4 网站脚本可能频繁重写标题

如果网页自身在某些时机不断重写标题，插件需要更稳妥的覆盖策略，否则可能出现闪回真实标题的情况。

## 10. 推荐实现路径

推荐采用两阶段落地：

### 阶段一：标题伪装 MVP

- 建立基础扩展结构
- 实现 popup 输入框
- 实现当前页面标题覆盖
- 实现当前 tab 内路由切换后的继续生效

### 阶段二：增强伪装能力

- 增加预设标题
- 增加 favicon 替换
- 增加按域名记忆规则
- 增加启停状态提示

## 11. 建议结论

从 Chrome 扩展能力边界来看，这个插件可以实现，而且适合先从一个非常小的 MVP 开始做。

最合理的技术路线是：

- 使用 Manifest V3
- 用 popup 收集用户输入
- 用 `activeTab` + `chrome.scripting` 动态注入当前 tab
- 用 content script 改写 `document.title`
- 用监听机制抵抗网页重写标题与 favicon

这条路线实现成本低、验证快，也最符合当前需求。

## 12. 当前假设

本方案基于以下假设编写：

- 首版目标网站为 YouTube。
- 首版优先解决标签标题暴露问题。
- 用户接受手动点击插件后设置伪装标题的交互方式。
- 自动化规则与多站点支持可以后续迭代。

如果后续确认要做“一键预设伪装”或“自动进入 YouTube 即套用固定标题”，需要再补充交互与存储规则设计。
